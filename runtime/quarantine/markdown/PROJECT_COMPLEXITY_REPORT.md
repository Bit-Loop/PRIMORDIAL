## State Machine Structure

### 1) Methodology Planner
- Core phases: `recon`, `analysis`, `exploitation`, `chaining`, `behavior_verification`, `memory_maintenance`, `notification` from `MethodologyPhase`.
- Per target, `_evaluate_target_methodology_state()` produces one state snapshot each tick with `phase`, `subphase`, `completion`, `candidate_actions`, `blockers`, `transition_reason`.
- Decision precedence: candidate actions, active tasks, no evidence bootstrap, compaction need, verified-interest count, then fallback backlog phases.
- Candidate generation is gated by intent, allowlists, evidence freshness, and required primitives, giving dense branch space.

### 2) Task Execution State
- `TaskStatus` states: `pending`, `running`, `waiting`, `succeeded`, `failed`, `blocked`, `needs_approval`, `cancelled`.
- Key transitions include:
  - `pending -> running` (claim), `pending -> waiting` (deferral), `pending -> needs_approval` (policy/gates), `pending -> blocked` (validation).
  - `running -> succeeded|failed`, with retry paths back to `pending` based on attempt budget.
  - `needs_approval -> pending|cancelled` via operator action.
  - recovery path: stale claims and execution exceptions can force rollback states.

### 3) Task Run State
- `TaskRunStatus` states: `claimed`, `running`, `succeeded`, `failed`, `timed_out`, `cancelled`.
- Mostly linear lifecycle with recovery override to `cancelled` for stale/non-finalized runs.

### 4) Session State
- `SessionStatus` states: `active`, `paused`, `completed`, `failed`.
- Runtime operations currently pivot on `active` and `paused` state transitions.

## State Machine Complexity
- Planner complexity is highest due to multi-file evidence/task scans and branching across candidate/action checks.
- Execution transitions are fewer but interleave scheduling, deferral, policy, and recovery logic each tick.
- Critical path is best described as branching-driven orchestration plus iterative retry recovery rather than deep algorithmic recursion.
